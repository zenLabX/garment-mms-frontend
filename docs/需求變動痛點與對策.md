# DDD 前端在需求持續變動下的痛點與對策

> 本文件回答一個問題：**需求複雜、變動性高、持續擴充時，這套架構（DDD 分層 + TanStack Query + 對照表驅動）憑什麼不崩塌？**
> 每條痛點都對應到 repo 內的具體機制與檔案，且各有一個可現場演練的範例（見文末「三級範例頁面」）。

## 一、九大痛點與機制級對策

### 1. 狀態機一直長：加一個狀態，列表 tag、詳情按鈕、可編輯性全要跟著變

- **崩塌路徑**：`if (status === ...)` 散落在列表、詳情、Tag 元件三處，加狀態靠全文搜尋，漏一處就是線上 bug。
- **對策**：「狀態 → 畫面行為」對照表 `Record<Status, View>`（範本：`src/domains/procurement/model/po.model.ts` 的 `poStatusViewMap`）。union type 加一個成員，**所有漏補的對照表在編譯期報錯**；元件只查表渲染，一行都不用改。
- **演練**：material 的分類加第四種 `Trim` → 只改 model 一行 + 假資料，列表/表單零改動（Select/Tag 都由 `Object.entries(viewMap)` 產生）。

### 2. 一筆資料改了，N 個畫面要同步

- **崩塌路徑**：每個頁面各自 refetch 或手動 setState，之後每加一個新畫面，都要回頭改所有會改到這筆資料的 mutation。
- **對策**：query key factory 的階層結構（`poKeys.all → lists() → list(params)`）+ mutation `onSuccess` 集中 invalidate（`po.api.ts`）。新畫面只要用同一個 factory 取 key，**自動被既有 invalidation 覆蓋**，mutation 端零改動。

### 3. 權限規則一直改：代理簽核、金額分級核決、部門例外……

- **崩塌路徑**：前端 `if (role === "manager" && amount < ...)` 與後端規則雙軌漂移，每次改規則要兩邊同步改。
- **對策**：DTO 能力旗標（`canApprove` 等）+ `CapabilityButton`；選單/路由走 `usePermittedMenu()`。規則只改在中台 Aggregate（mock 中為 `mocks/data.ts` 的 `computeCapabilities`），**前端零改動**。

### 4. 跨 domain 連動：採購核准 → 庫存看板、待辦數字要跟著刷新

- **崩塌路徑**：元件裡到處塞 `refetch()` 或自建 event bus，半年後沒有人能盤點出「核准到底會連動幾個畫面」。
- **對策**：跨模組 invalidation **集中宣告在 mutation 所屬的 `*.api.ts`**（如 `po.api.ts` 的 `usePoAction.onSuccess` 加一行 `invalidateQueries(inventoryKeys.dashboards())`）。要盤點連動關係，grep 一個 keys 名稱就結束。
- **耦合方向規範**：**寫入方（command 端）import 讀取方（read model 端）的 keys 物件**，且只 import keys，不 import hooks / 元件。

### 5. 需求加欄位

- **崩塌路徑**：手寫型別忘了改，執行期才炸。
- **對策**：接通中台後 `npm run gen:api` 產 DTO、`*.model.ts` 改 re-export。改欄位 → 重產型別 → **tsc 在編譯期列出所有要改的點**，型別驅動修改。骨架階段的手寫型別同樣集中在 `model/`，zod schema 與型別同源（`z.infer`），改一處全連動。

### 6. 元件被業務邏輯污染，逐漸不可複用

- **崩塌路徑**：Tag / 按鈕元件裡長出 `if (status === "X" && user.role === "Y")`，最後每個 domain 都複製一份改。
- **對策**：業務知識只存在兩個地方——`model/` 的純資料對照表、中台回傳的旗標。`StatusTag` / `CapabilityButton` / `AppTable` / `EChart` 全是**無業務知識的殼**，任何 domain 直接複用。

### 7. Fat store 崩塌（舊專案的教訓）

- **崩塌路徑**：API 回應 copy 進全域 store，store 越長越大；資料何時過期、被誰改動無人知道，頁面間互相污染。
- **對策**：紀律（CLAUDE.md 第 1 條）——server state 只存在 TanStack Query 快取，Zustand 只放 UI state（篩選、分頁位置、多步驟表單草稿）。過期 / 重抓 / 去重 / 競態全交給 query cache。
- **灰色地帶判準**（多步驟表單草稿）：**還沒送出 = UI state（Zustand）；送出成功那一刻起 = server state（Query），立即 `resetDraft()`**，之後畫面一律從 Query 讀。

### 8. 資料量成長：上線時 30 筆前端過濾沒事，兩年後 3 萬筆

- **崩塌路徑**：抓全量前端過濾的頁面，隨資料成長逐一爆炸重寫。
- **對策**：紀律（第 3 條）——`AppTable` 的 props 型別**強制**收 `PagedResult<T>`，從第一天就是伺服器端分頁 + `placeholderData: keepPreviousData`。資料量成長不需要重寫任何頁面。

### 9. 即時性需求後加：「看板要即時」永遠是上線後才提

- **崩塌路徑**：自建 WebSocket 資料同步層，推播 payload 與 REST 回應兩套資料形狀，雙軌維護。
- **對策**：`shared/realtime/realtime-client.ts` 的原則——**推播只當「快取失效通知」**（收到事件就 `invalidateQueries`），資料一律重新向中台要。從輪詢（`refetchInterval`）升級到 SignalR 時，**各 domain 的 query / 元件零改動**，只換 client 實作（事件 → 失效的對照集中在 `app/realtime-handlers.ts`）。

## 二、誠實的代價與緩解

這套架構不是免費的。以下是它的稅，以及為什麼值得付：

| 代價 | 說明 | 緩解 |
|---|---|---|
| **樣板碼多** | 每個 domain 都要 keys / hooks / model / store 四件套，小功能也逃不掉 | 接受這筆「樸素稅」：每個 domain 長得一模一樣，新人照抄 `procurement/` 即可。**不要**為了消樣板建 hooks 工廠——那只是把複雜度搬到更難懂的地方 |
| **對照表可能爆炸** | 狀態 × 角色 × 頁面三個維度以上時，單一 Record 會變超寬 | 對照表只放「狀態 → 畫面」一個維度；角色維度已外移到伺服器能力旗標（天然砍掉一維）。某頁行為特殊時為該頁建第二張小表，而不是把一張表加寬到不可讀 |
| **msw 假資料是雙份邏輯** | mock 的 `computeCapabilities`、handlers 要跟著中台規則演進 | mock 定位為「離線 demo + 測試」，接通中台後整包 `mocks/` 移除。新 domain 的假資料從單一來源派生（讀模型由 `poDb` / `reqDb` / `materialDb` 算出來），壓低同步面積 |
| **invalidate lists() 較粗放** | 會標記所有分頁組合為 stale，理論上有浪費 | TanStack 只主動重抓 active query + staleTime 30s，實務夠用；真成瓶頸再對單一 key 精準 invalidate，屬局部優化不動架構 |
| **跨 domain import 有耦合** | `po.api.ts` invalidate `inventoryKeys` 需要 import 別的 domain | 規範單向：寫入方 import 讀取方的 **keys 物件**，禁止 import hooks / 元件。連動全貌 grep 一次可盤點 |
| **樂觀更新有風險** | UI 先變、API 後到，失敗要回滾 | 取捨標準寫死：**前端能 100% 預測結果的低風險動作才做 optimistic**（如撤回 → 必回 Draft）；簽核類結果由中台計算（旗標、歷程），前端猜不準，一律保守 invalidate |

## 三、三級範例頁面（本 repo 內可跑的證明）

| 級 | domain / 頁面 | 展示的對策 |
|---|---|---|
| 低 | `material` 物料主檔：列表 + 搜尋 + 新建/編輯表單 | 對照表（#1）、key factory（#2）、zod 邊界（#5）、伺服器分頁（#8）、新 domain 照抄成本極低 |
| 中 | `requisition` 領料申請：三步驟表單 + 依賴查詢 + 樂觀更新 | UI/server state 分界（#7）、`enabled` 依賴查詢與跨 domain 快取共享、超領由伺服器 422 把關（#3 的變體）、樂觀更新取捨 |
| 高 | `inventory` 庫存看板：多卡多圖 + 輪詢 + 跨 domain 連動 | 跨 domain invalidation（#4）、全域篩選器驅動整版、輪詢→SignalR 升級路徑（#9）、讀模型從交易資料派生 |

**殺手級 demo**：開兩個分頁，用 Lisa 身分核准一張採購單 → 庫存看板的「在途採購量」卡片與分類圖自動更新。做到這件事只需要在 `po.api.ts` 的 `onSuccess` 加**一行** invalidate——這就是「連動不散落」。
